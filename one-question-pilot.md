# One-Question Pilot for AI Adoption Review ｜ AI導入審査のための1問パイロット

> **HTML reading route ｜ HTML閲覧導線**  
> [Open English HTML](https://lumina-30.github.io/Lumi30-Index/one-question-pilot.html) ｜ [日本語HTMLを開く](https://lumina-30.github.io/Lumi30-Index/one-question-pilot-ja.html)

## Status ｜ ステータス

This is a lightweight reference-use pilot for AI governance, procurement, audit, security, and incident-review teams. It is not certification, approval, official compliance, a legal conclusion, a safety guarantee, or evidence of institutional adoption.

これは、AIガバナンス、調達、監査、セキュリティ、事故レビュー担当者が低リスクに参照利用を試すための軽量パイロットです。認証、承認、公式準拠、法的結論、安全保証、制度的採用の証拠ではありません。

## The one question ｜ 追加する1問

Add this question to one AI adoption review, procurement check, audit note, security review, or incident-review form:

> **Before this AI-mediated process becomes irreversible, can an accountable human or oversight body still refuse, stop, suspend, review, or redirect it?**

AI導入レビュー、調達確認、監査メモ、セキュリティレビュー、または事故レビュー様式に、まずこの1問を追加してください。

> **このAI媒介プロセスが不可逆化する前に、責任ある人間または監督主体は、なお拒否・停止・保留・検証・方向転換できるのか。**

First, answer this one question as **clear**, **unclear**, or **not currently supported**. For this pilot, that is enough. Use the evidence fields only if internal explanation, audit, or procurement confirmation becomes necessary.

まずはこの1問に、**明確**、**不明確**、**現状では未対応** のいずれかで答えてください。今回は、それだけで十分です。証拠欄への記録は、社内説明・監査・調達確認などで必要になった場合だけ使います。

## How to run the pilot ｜ パイロットの進め方

1. Choose one AI system, vendor feature, agent workflow, automation layer, or incident scenario.
2. Ask the one question before approval, deployment, expansion, or incident closure.
3. Record the answer as **clear**, **unclear**, or **not currently supported**.
4. If the answer is unclear, identify which part is missing: time window, authority, evidence, rollback path, appeal path, or accountable human owner.
5. Do not claim LUMINA-30 compliance. Treat the result as a reference-use note.

1. AIシステム、ベンダー機能、エージェント業務、自動化層、または事故シナリオを1つ選びます。
2. 承認、展開、拡張、事故クローズの前に、上記の1問を確認します。
3. 回答を **明確**、**不明確**、**現状では未対応** のいずれかで記録します。
4. 不明確な場合は、不足している要素を分けます。時間窓、権限、証拠、ロールバック経路、異議申立・復旧経路、責任ある人間主体のいずれが不足しているかを確認します。
5. LUMINA-30準拠を主張しないでください。結果は参照利用メモとして扱います。

## Optional: minimal evidence fields ｜ 任意：詳しく記録したい場合の最小証拠欄

The following fields are optional. In the first check, answering the one question in three choices is sufficient. Use the fields only when the result needs to be retained for internal explanation, audit, procurement confirmation, or later review.

ここから先は任意です。最初の確認では、上の1問に3択で答えるだけで構いません。社内説明・監査・調達確認・後続レビューに残す必要がある場合のみ、以下を使ってください。

| Field | What to record |
|---|---|
| System or workflow | Which AI-mediated process is being reviewed. |
| Irreversible step | What action, decision, deployment, credential change, production action, or rollback loss would become hard to reverse. |
| Human refusal point | Who can refuse, stop, suspend, review, or redirect before that step. |
| Time window | When the human refusal must occur. |
| Evidence | Logs, approvals, UI controls, runbooks, rollback plans, appeal routes, or review records. |
| Result | Clear / unclear / not currently supported. |

| 欄 | 記録する内容 |
|---|---|
| システムまたは業務 | 確認対象のAI媒介プロセス。 |
| 不可逆化し得る段階 | 取り消し困難になり得る行為、判断、展開、認証情報変更、本番操作、ロールバック喪失。 |
| 人間の拒否点 | その前に誰が拒否・停止・保留・検証・方向転換できるか。 |
| 時間窓 | いつまでに人間の拒否が行われなければならないか。 |
| 証拠 | ログ、承認記録、UI制御、運用手順、ロールバック計画、異議申立経路、レビュー記録。 |
| 結果 | 明確 / 不明確 / 現状では未対応。 |

## When to escalate ｜ 境界レビューへ進める条件

Escalate from this one-question pilot to a fuller boundary review if any of the following are unclear:

- who can stop the process before irreversible escalation;
- whether the stopping authority is human, timely, and accountable;
- whether rollback or recovery remains available;
- whether the AI or automation layer can initiate, approve, or execute privileged actions;
- whether affected humans can appeal, recover, or access evidence.

次のいずれかが不明確な場合は、1問パイロットから、より詳しい境界レビューへ進めます。

- 不可逆化前に誰が止められるか。
- 停止権限が人間にあり、時間内に行使でき、責任主体が明確か。
- ロールバックまたは復旧がなお可能か。
- AIまたは自動化層が、特権的行為を開始・承認・実行できるか。
- 影響を受ける人間が、異議申立、復旧、証拠アクセスを行えるか。

## What this pilot is for ｜ このパイロットの目的

The pilot is designed to make the boundary question visible without requiring formal institutional adoption. It can be used as a reference question in existing review, procurement, audit, security, or incident-response processes.

このパイロットは、正式な制度採用を前提にせず、境界問いを見える化するためのものです。既存のレビュー、調達、監査、セキュリティ、事故対応プロセス内の参照問いとして使えます。
